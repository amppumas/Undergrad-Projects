Métodos ágiles

A medida que las empresas notaron la utilidad de los sistemas computacionales se vieron en la necesidad de implementarlos para mantenerse competitivos. Pero dado el ambiente empresarial tan volátil en que se vive las estrategias comunes a la ingeniería, esto es, la planeación detallada de un proyecto, no son compatibles con este mercado tan cambiante. Es por ello que surgió un acercamiento diferente para resolver este problema.

Los métodos ágiles permiten crear sistemas que se adapten al cambio de manera fluida, permiten un acercamiento con el cliente y además son compatibles con el desarrollo acelerado que requiere la industria empresarial. Agregado a esto, los requerimientos cambian constantemente o simplemente no se conocen a priori.

Estos métodos, cabe aclarar, no son recomendados para sistemas críticos o con equipos de desarrollo muy grandes, ya que en el primer caso se requiere de una alta fiabilidad y seguridad que no ofrece este metodo, ademas los requerimientos de un sistema crítico casi siempre están dados desde el principio. Para el segundo caso, es difícil que un equipo de programadores muy grande pueda colaborar sin interferir y conociendo todo el sistema. Pues como se ahondará más adelante es necesario que todo el equipo de trabajo sepa el funcionamiento de todo el código y pueda modificarlo a discreción.

Los métodos ágiles más usados en la actualidad son el método de programación extrema y el método scrum. Ambos son variaciones del método incremental, pero en lugar de estar basados en un plan se van adaptando al cambio. No se desarrollan de una pasada, sino que se van dividiendo en módulos y se van agregando en forma de incrementos. En general se puede decir que los métodos ágiles comparten las siguientes características.

  1. Las etapas de especificación, diseño e implementación están relacionadas, la documentación se minimiza y los requerimientos no son detallados.

  2. El sistema es versionado. Es decir, cada que se libera el sistema se agrega una nueva funcionalidad o se realiza una documentación del sistema. Los clientes retroalimentan cada una de estas etapas.

  3. Las interfaces de usuario se hacen adaptativas, es decir, agregar funcionalidades no supone un rediseño completo ni una nueva implementación.

Además de compartir estas características los métodos ágiles, en su mayoría toman como principios lo que se conoce como Manifiesto Ágil y que en resumen tiene los siguientes conceptos:

Participación del cliente: Para que el software se diseñe a medida del cliente este debe estar presente durante el proceso de desarrollo y ser colaborativo, es decir que evalúe puntualmente las liberaciones del sistema.
Entrega Incremental: Ya que está basado en el método incremental es natural que se desarrolle de esta manera, y en cada uno es indispensable que el cliente fije los requerimientos que deberá tener la siguiente liberación.
Personas, no procesos: A diferencia de los métodos tradicionales, que son más centrados en los procesos, en el método ágil se debe de enfatizar en el aprovechamiento de los recursos humanos disponibles y sin entorpecer su ritmo de trabajo.
Adoptar el cambio: En cuanto se cambien los requerimientos del sistema, este debe adaptarse al cambio y adoptar estos nuevos requerimientos.
Mantener simplicidad: Mientras se desarrolla el proyecto la simplicidad y el codigo de alta calidad deben ser prioridades para el equipo de desarrollo.

Sin embargo, hay que tomar en cuenta ciertos aspectos antes de elegir un método de desarrollo ágil.

  1. Se debe tener un cliente dispuesto a la colaboración y que sea representativo del usuario final del sistema.

  2. El equipo de desarrollo debe estar acostumbrado a este ritmo de trabajo y también debe estar dispuesto a colaborar con el cliente y con sus demás compañeros.

  3. Mantener el código simple requiere de mucho esfuerzo y dedicación del equipo de desarrollo y bajo la presión de las fechas, en la práctica esto se omite.

  4. Es difícil para una compañía tradicional cambiar su método a uno más flexible.

Además de estos problemas, los métodos ágiles tienen un problema más general. Los contratos de desarrollo de software normalmente se basan en los requerimientos del sistema y en los métodos ágiles esto no se conoce de antemano.

Este método además no está formalizado para el mantenimiento de sistemas, y aunque se ha utilizado es más parecido al método de evolución de software. Además, ya que la documentación es mínima es difícil dar mantenimiento a estos sistemas, por eso se recomiendan estos métodos para sistemas de un ciclo de vida relativamente bajo. En este mismo rubro es difícil que el cliente se involucre cuando requiere mantener un sistema que cuando lo quiere nuevo. Y relacionado con los equipos de trabajo estos no se pueden modificar ya que se han integrado y conocen a la perfección la estructura del sistema.
